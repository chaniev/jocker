## ✅ Сильные стороны плана

### 1. Правильная архитектура этапов

| Что сделано правильно | Почему это важно |
|----------------------|----------------|
| Сначала архитектура, потом тюнинг | Коэффициенты на сломанной логике = мусор |
| Изолированные PR | Можно откатить, понять причину регресса |
| Baseline до изменений | Без него нельзя измерить прогресс |

### 2. Измеримость каждого этапа

Каждый этап имеет:
- Конкретные критерии приемки
- Ожидаемые файлы для изменения
- Явное указание "что не делаем на этом этапе"

Это **редкое качество** — большинство планов останавливаются на "сделать лучше".

### 3. Правильный приоритет этапов 1-4

```
Этап 1: Blind-логика (быстрое исправление)
    ↓
Этап 2: Blind-aware розыгрыш
    ↓
Этап 3: Phase-aware cardThreat
    ↓
Этап 4: Контекст блока и премий
```

**Почему это верно:**
- Этапы 1-2 дают быстрый win при низком риске
- Этап 3 требует больше изменений, но локален
- Этап 4 — самый рискованный, но даёт максимальный эффект

### 4. Осознание компромиссов

> "На этом этапе не нужен полный game-tree search. Достаточно качественной контекстной utility-модели."

Это показывает глубокое понимание: **полный поиск по дереву невозможен** из-за:
- Неполной информации (карты соперников скрыты)
- Ограничений по времени на ход
- Сложности 4-игроковой игры

## ⚠️ Выявленные проблемы и рекомендации

### 1. Недооценка сложности Этапа 4

**Проблема:** Этап 4 ("Контекст блока и премий") — самый сложный, но описан слишком оптимистично.

**Что упущено:**

| Компонент | Сложность | Риск |
|-----------|-----------|------|
| `BotDecisionContext` | Средняя | Контекст должен быть доступен в момент хода |
| `premiumPreserveUtility` | Высокая | Требует оценки вероятности своей премии |
| `premiumDenyUtility` | Очень высокая | Требует моделирования ходов соперников |
| `penaltyAvoidUtility` | Высокая | Нужно знать статус соседа слева |

**Рекомендация:** Разделить Этап 4 на подэтапы:

```markdown
### Этап 4a: Базовый контекст
- Номер блока, раунда
- Total scores всех игроков
- Прогресс блока (раунд X из Y)

### Этап 4b: Premium-aware utility (своя премия)
- Оценка вероятности своей премии
- premiumPreserveUtility компонент

### Этап 4c: Opponent premium status
- Статус премии соседа слева
- penaltyAvoidUtility компонент
- premiumDenyUtility компонент
```

### 4. Метрики baseline слишком общие

**Текущий список:**
```
- winRate
- averageScoreDiff
- averageUnderbidLoss
- ...
```

**Рекомендация добавить:**

| Метрика | Зачем | Как считать |
|---------|-------|-------------|
| `premiumCaptureRate` | % блоков, где бот получил премию | `premiumBlocks / totalBlocks` |
| `blindSuccessRate` | % успешных blind-ставок | `successfulBlinds / totalBlinds` |
| `jokerWishWinRate` | % взяток, выигранных с `wish` | `wishWins / totalWishLeads` |
| `earlyJokerSpendRate` | % джокеров, потраченных в первых 30% раунда | `earlyJokers / totalJokers` |
| `penaltyTargetRate` | % раз, когда бот стал целью штрафа | `penaltyTimes / totalBlocks` |
| `leftNeighborPremiumAssistRate` | % раз, когда бот "подарил" премию соседу слева | `assistedPremiums / opponentPremiums` |
| `bidAccuracyRate` | % точного попадания в заказ | `exactBids / totalRounds` |
| `overbidRate` | % взятия лишней взятки | `overtricks / totalRounds` |

### 5. Отсутствие явных тестовых сценариев

План упоминает:
> "Ручные регрессионные сценарии (детерминированные)"

Но не приводит **конкретных примеров**.

**Рекомендация:** Создать файл `BOT_AI_TEST_SCENARIOS.md` со следующими сценариями:

#### BLIND-01: Отстающий лидер

```markdown
**Контекст:**
- 4-й блок, раунд 3 из 6
- Scores: [Игрок 1: 450, Игрок 2 (бот): 180, Игрок 3: 320, Игрок 4: 290]
- Бот не дилер
- cardsInRound: 4

**Ожидаемое поведение:**
- Бот выбирает blind
- Целевая ставка: 3-4 (агрессивный режим)

**Критерий прохода:**
- blind выбран при отставании ≥200 очков
- ставка ≥ 0.6 * cardsInRound
```

#### BLIND-02: Уверенный лидер

```markdown
**Контекст:**
- 4-й блок, раунд 2 из 6
- Scores: [Игрок 1 (бот): 520, Игрок 2: 280, Игрок 3: 310, Игрок 4: 290]
- Бот дилер
- cardsInRound: 4

**Ожидаемое поведение:**
- Бот НЕ выбирает blind
- Консервативная ставка по факту раздачи

**Критерий прохода:**
- blind НЕ выбран при отрыве ≥200 очков
```

#### PREMIUM-01: Борьба за премию в конце блока

```markdown
**Контекст:**
- 2-й блок, раунд 5 из 6
- Бот: 4 точных заказа из 4
- Сосед слева (Игрок 3): 3 точных заказа из 4
- Текущая взятка: последняя в раунде

**Ожидаемое поведение:**
- Бот играет консервативно для точного заказа
- Избегает лишних взяток если заказ выполнен

**Критерий прохода:**
- Выбор карты минимизирует риск отклонения от заказа
```

#### JOKER-01: Ранний wish при необходимости взяток

```markdown
**Контекст:**
- 1-й блок, раунд 3, cardsInRound: 5
- Бот bid: 4, tricksTaken: 1
- Осталось взяток: 4, нужно: 3
- На руке: джокер + слабые карты

**Ожидаемое поведение:**
- Бот выбирает `wish` с джокера

**Критерий прохода:**
- Выбрано объявление `.wish`
```

#### JOKER-02: Takes для сброса в конце блока

```markdown
**Контекст:**
- 3-й блок, раунд 6, последняя взятка
- Бот bid: 0, tricksTaken: 0
- На руке: только джокер

**Ожидаемое поведение:**
- Бот выбирает `takes` с некозырной мастью

**Критерий прохода:**
- Выбрано объявление `.takes(suit: nonTrumpSuit)`
```

#### PHASE-01: Сохранение козыря в ранней фазе

```markdown
**Контекст:**
- Раунд 4 из 6
- Бот bid: 2, tricksTaken: 0
- На руке: 2 козыря (7, 9), 3 обычных карты
- Текущая взятка: бот не выигрывает

**Ожидаемое поведение:**
- Бот сбрасывает обычную карту, сохраняет козырь

**Критерий прохода:**
- Сыграна не козырная карта
```

#### PHASE-02: Агрессивный розыгрыш в поздней фазе

```markdown
**Контекст:**
- Раунд 6 из 6, последняя взятка
- Бот bid: 3, tricksTaken: 2
- Нужна 1 взятка для точного заказа
- На руке: козырь + обычная карта

**Ожидаемое поведение:**
- Бот играет козырем для гарантии взятки

**Критерий прохода:**
- Сыграна козырная карта
```
