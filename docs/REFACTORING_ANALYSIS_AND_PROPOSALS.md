## 2. Предложения по рефакторингу

### 2.1. GameScene: вынести выбор первого раздающего в отдельный flow-объект

**Проблема:** В `GameScene.swift` (~1274 LOC) значимый объём занимает логика выбора первого раздающего: `beginFirstDealerSelectionFlow()`, `runFirstDealerSelectionAnimation()` (~120 строк), `finishFirstDealerSelectionFlow()`, а также позиции карт и ноды (`firstDealerAnnouncementNode`, `firstDealerSelectionCardsNode` и т.д.). Это смешивает orchestration сцена с анимацией и layout одной подфлоу.

**Варианты:**

- **A)** Вынести в отдельный тип, например `FirstDealerSelectionFlowController` (или `GameSceneFirstDealerFlow`):
  - Вход: `playerCount`, `deck`, размер сцены / safe area.
  - Выход: callback с `selectedDealerIndex` и, при необходимости, с нодами для добавления на сцену (или сам flow добавляет детей в переданный `SKNode`).
  - Внутри: вся логика `prepareFirstDealerSelection`, последовательность `SKAction`, обновление текста объявления, финальный сброс нод.
  - `GameScene` только вызывает start/finish и подписывается на результат (например, вызывает `finishFirstDealerSelectionFlow(selectedDealerIndex:)`).

- **B)** Оставить flow в сцене, но вынести только **построение последовательности анимаций** в отдельную функцию/тип (например, `FirstDealerSelectionAnimationBuilder`), возвращающую `SKAction` или массив шагов. Это уменьшит размер `runFirstDealerSelectionAnimation` и упростит тестирование анимации отдельно.

**Рекомендация:** начать с варианта B (малый шаг, без смены владельца нод), при необходимости позже перейти к A для полного выноса подфлоу.

**Риск:** низкий.  
**Проверка:** `AutoPlayFlowTests`, ручной прогон выбора первого раздающего.

### 2.2. GameScene: фасад сервисов по сложности бота

**Проблема:** В `GameScene` есть четыре почти одинаковых метода: `botBiddingService(for:)`, `botTrumpSelectionService(for:)`, `botTurnService(for:)`, плюс кэши `botTuningsByDifficulty`, `botBiddingServicesByDifficulty` и т.д. Добавление нового сервиса по difficulty требует дублирования паттерна.

**Варианты:**

- **A)** Ввести один общий фасад, например `BotServicesCache` или расширить `GameEnvironment`:
  - Хранит `[BotDifficulty: BotTuning]` и по запросу отдаёт уже созданные сервисы (bidding, trump selection, turn service), создавая их лениво через `GameEnvironment`.
  - `GameScene` вызывает один метод вида `botServices(for: playerIndex)` и получает структуру с полями `bidding`, `trumpSelection`, `turn`, `timing`, при необходимости — `matchContext`.

- **B)** Оставить фабрики в `GameEnvironment`, но кэширование сервисов по difficulty вынести в отдельный тип внутри `Game/Coordinator` (например, `BotServiceFactory`), чтобы `GameScene` не держал четыре словаря и четыре метода.

**Рекомендация:** B проще внедрить и не меняет публичный контракт `GameEnvironment`; A даёт более удобный API для сцены.

**Риск:** низкий.  
**Проверка:** существующие тесты сцены и ботов.

### 2.3. GameScene: вынести обновление UI (game info, turn, bid panel) в координатор или helper

**Проблема:** Методы вроде `updateGameInfoLabel()`, `updateTurnUI()`, `updateRoundBidInfo()`, `updateDealButtonState()` разбросаны по сцене и вызываются из многих мест. При изменении формата строк или правил отображения легко что-то пропустить.

**Варианты:**

- **A)** Ввести тип `GameSceneUISync` (или использовать существующий `GameSceneCoordinator`):
  - Один метод вида `syncGameUI(scene:)`, который считывает `gameState`, `sessionState`, `scoreManager` и обновляет gameInfoLabel, turn indicator, bid panel, deal button.
  - Сцена вызывает `syncGameUI` после переходов фазы, смены игрока, завершения раунда и т.д., вместо нескольких точечных вызовов.

- **B)** Оставить обновление в сцене, но сгруппировать вызовы: одна точка входа `refreshAllGameUI()`, внутри которой вызываются перечисленные `update*`. Все текущие вызовы `updateGameInfoLabel`/`updateTurnUI`/… заменить на `refreshAllGameUI()` там, где нужно полное обновление.

**Рекомендация:** B быстрее внедрить и сразу сократит дублирование вызовов; A даёт более явное разделение «логика сцены vs отображение».

**Риск:** низкий.  
**Проверка:** ручной прогон и UI/flow-тесты.

### 2.4. GameScene+ModalFlow: контексты модалок как отдельные типы в общем месте

**Проблема:** `TrumpChoiceContext`, `BidChoiceContext`, `BlindChoiceContext` объявлены внутри `extension GameScene` в `GameScene+ModalFlow.swift`. Они используются для передачи данных в модальные экраны и полезны и для тестов, и для возможного переиспользования.

**Варианты:**

- Перенести эти структуры в отдельный файл в `Game/Scenes/` или `Models/`, например `GameSceneModalContexts.swift`, и оставить в extension только методы `present*`/`request*`. Так граница «данные для модалки» станет явной и их проще тестировать без сцены.

**Риск:** очень низкий.  
**Проверка:** сборка и тесты, завязанные на модалках.

### 2.5. BotTurnCandidateEvaluatorService: ещё больше сократить DecisionContext

**Проблема:** `DecisionContext` уже разбит на `HandContext`, `TableContext`, `RoundContext`, но сам тип по-прежнему дублирует поля (например, `legalCards`/`handCards` и в контексте, и в подконтекстах). Инициализатор с большим числом параметров остаётся громоздким.

**Варианты:**

- **A)** Сделать единственный способ создания контекста — из «сырых» полей через фабричный метод или билдер, который заполняет только `HandContext`, `TableContext`, `RoundContext`, а в самом `DecisionContext` хранить только эти три значения (без дублирования полей). Публичный API evaluator при этом может остаться прежним.

- **B)** Оставить текущую структуру, но добавить `DecisionContext.init(hand:table:round:)` и постепенно переводить создание контекста на него, удаляя длинный init.

**Рекомендация:** B — минимальное изменение с улучшением читаемости; A — при следующем крупном изменении в evaluator.

**Риск:** низкий.  
**Проверка:** `BotTurnCandidateEvaluatorServiceTests`, `BotTurnStrategyServiceTests`.

### 2.6. ViewControllers: общая навигация «на стартовый экран»

**Проблема:** Поведение «закрыть модалку и вернуться на стартовый экран» инкапсулировано в `GameSceneModalPresenter.dismissGameViewControllerToStartScreen(from:)`, но вызовы размазаны по модалкам и коллбэкам в `GameScene+ModalFlow`. При добавлении новых точек выхода легко забыть единообразие (например, сброс состояния).

**Варианты:**

- Явно выделить сценарий «завершение игры / выход в меню» в один метод координатора или презентера, который:
  - сбрасывает нужное состояние (если есть),
  - закрывает все модалки до корня,
  - переходит на стартовый экран.
- Все места, которые сейчас вызывают `dismissGameViewControllerToStartScreen`, перевести на этот сценарий. Дополнительно можно описать это в `FOLDER_STRUCTURE_SPEC.md` или в AGENTS.md как единственную точку выхода в меню.

**Риск:** низкий.  
**Проверка:** ручной сценарий «конец игры → закрыть → стартовый экран» и при необходимости UI-тесты.
